27    /**
28    * Provides file-system based storage for torrent files.
29    *
30    * <p>Information about the collection of files in a torrent is stored as a list of strings in the torrent metainfo.
31    * This makes it possible for malicious parties to create torrents with malformed pathnames,
32    * e.g. by using relative paths (. and ..), empty and invalid directory and file names, etc.
33    * In the worst case this can lead to loss and corruption of user data and execution of arbitrary code.
34    *
35    * <p>This storage implementation performs normalization of file paths, ensuring that:
36    * <ul>
37    * <li>all torrent files are stored inside the root directory of this storage (see {@link #FileSystemStorage(File)})</li>
38    * <li>all individual paths are checked for potential issues and fixed in a consistent way (see below)</li>
39    * </ul>
40    *
41    * <p><b>Algorithm for resolving paths:</b><br>
42    * 1) The following transformations are applied to each individual path element:
43    * <ul>
44    * <li>trimming whitespaces,</li>
45    * <li>truncating trailing dots and whitespaces recursively,</li>
46    * <li>substituting empty names with an underscore character
47    * (this also includes names that became empty after the truncation of whitespaces and dots),</li>
48    * <li>in case there is a leading or trailing file separator,
49    * it is assumed that the path starts or ends with an empty name, respectively.</li>
50    * </ul><br>
51    * 2) Normalized path elements are concatenated together using {@link File#separator} as the delimiter.
52    *
53    * <p><b>Examples:</b><br>
54    * {@code "a/b/c" => "a/b/c"}<br>
55    * {@code " a/ b /c" => "a/b/c"}<br>
56    * {@code ".a/.b" => ".a/.b"}<br>
57    * {@code "a./.b." => "a/.b"}<br>
58    * {@code "" => "_"}<br>
59    * {@code "a//b" => "a/_/b"}<br>
60    * {@code "." => "_"}<br>
61    * {@code ".." => "_"}<br>
62    * {@code "/" => "_/_"}<br>
63    * {@code "/a/b/c" => "_/a/b/c"}<br>
64    *
65    * @since 1.0
66    */
